Process

  • Parallel Development - Work in the context of real content, not style Guide
  • Application -> Toolkit -> Style Guide (Style guide first is typically not realistic)

Groundwork

  • Set the stage for what will be - base styles
  • Then organize Components
  • Provide Settings
  • Add components as needed, not as wanted
  • Don't try to force too much.
  • alerts, buttons, dropdowns, forms, media, tables, tooltips, etc

Rinse and Repeat

  • Watch for duplicate code, then refactor
  • adjust as needed
  • Create new variables - maps

Implementation

Single Responsibility Principle

  • Every class should have a single responsibility
  • Responsibility should be encapsulated

Decouple HTML and CSS

  • Remove parent container dependency
  • Allow elements to adapt

Keep Specificity Low

  • Be explicit
  • Use Classes
  • Avoid nested selectors

Name with care

  • Name for understanding - so developers will know what is going on
  • Favor functional names

Intelligently Use Extends

  • Extend common property and value pairs
  • Favor placholder extends
  • Don't chain extends
  • Use a sensible pattern

Leverage maps

  • Dynamically generate styles
  • Easily write extends
  • Quickly build components

Watch Your Output

  • Test early, test often
  • Monitor speed and performance
  • Concatenate & compress files

Measure your Improvements

  • Identify a baseline
  • Select a few KPIs (File size, # of rulesets, # of properties like float)
  • Look into stylestats, csscss, and csslint

Final Thoughts

  • Dig into the Sass documentation - functions, etc.
  • Learn your environment - Ruby, etc.

Don't Overthink

  • Write code out the long way, then refactor
  • Rename, rewrite, and refactor willingly
  • Keep an eye on output (Sassmeister)
  • Document everything! (How, What, and Why)

Work Together

  • Seek help and get feedback
  • Open the style framework to everyone
  • Get feedback from Twitter
